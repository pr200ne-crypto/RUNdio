**0.2 での主な変更（要約）**: 課題提出フェーズの**主クライアントを PC ブラウザ（Web）**に固定。地図は **Leaflet + OpenStreetMap タイル**（利用条件・帰属表示の遵守）、施設は **JSON モック**優先。走行中は**実 GPS 必須とせず**、**タイマー／プログレスバーによる疑似ラン**でデモ可能とする。音声は **Web Speech API や短いダミー音声**を優先し、有料 TTS・有料 Maps は避ける。

## システム概要

**ランディオ**は、ソロランナー向けに「走る前のプランニング」「走行中のパーソナル音声体験（ランニング用ラジオ）」「走後の振り返りと次回提案」を一続きで提供するシステムである。解決する課題は、一人ランにおける練習のマンネリ、モチベーション維持、走行中の楽しさの不足である。目標とする成果は、セッション完了率および主観的な伴走感・満足度の向上である（数値目標は別途設定）。期待されるビジネスインパクトは個人向け継続課金やフリーミアム等を想定するが、収益モデルは未確定である。

**課題・学習・提出用の当面の形態**は、**PC 上のブラウザ**でサービスを提示し、併せてスライドで説明するデモである（`docs/input/README.md`）。このため **MVP デモ実装の主戦場は Web（例: Next.js）** とし、Expo／ネイティブは「実機アプリが必須」となった場合の拡張として位置づける。

主な機能は、(1) ルート定義とルート上距離に基づく条件検索によるチェックポイント選定、(2) 距離または時間をユーザーが選べる区切りに応じた音声フィードバック（数値・励まし・オープニング／中盤／終盤の番組的構成）、(3) 他オーディオ（ラジオ等）との共存（ダッキング等）— **Web デモではブラウザの制約に合わせて簡略化可**、(4) 走後サマリーと次回セッション提案、(5) 将来の Google Fit 等連携のための拡張ポイントである。差別化は「計画・音声・走後」が同一体験として設計される点と、自然な人間的な音声品質を最優先の体験目標に置く点にある。

ユーザーにとっての直接的価値は、走る理由と流れが明確になり、走行中に退屈しにくいことである。実現の概要として、**デモ版**ではブラウザの**時間計測・疑似距離進行・音声出力**を用い、**モックまたは静的データ**による条件検索を組み合わせる。将来の本番想定ではモバイル端末の位置情報と外部施設 API を組み合わせうる。スケーラビリティは初期ユーザー規模を小さく想定し、API コストとバックグラウンド処理の最適化を段階的に行う。

## 機能要件

### F-1 走前プランニング

- ユーザーは**今日のラン用ルート**をアプリ上で定義または選択できること。  
- **ルート上の前方距離**を基準に、**施設カテゴリ等の条件**で候補を検索し、チェックポイント（例：ゴール付近の入浴施設）を選択できること。MVP では少なくとも **1 カテゴリ**を対象とする（具体カテゴリは実装時に確定）。  
- 目標距離または目標時間（またはフリー）を設定できること（詳細は入力仕様で確定）。 [仮]

### F-2 走行中セッション

- **経過時間およびルート上の進捗距離（相当）**を記録すること。**Web デモ（版 0.2）**では、実 GPS に代えて**タイマー連動・一定ペース仮定・プログレス操作**等による**疑似進行**でよい。将来のモバイル本番では、GPS（および OS が許容する手段）による**走行軌跡**記録を想定する。  
- ユーザーは**区切り基準**として**距離**または**時間**のいずれかを選択でき、選択に応じて音声イベントが発火すること。  
- 音声は **数値フィードバック**（ペース、可能なら心拍等）と**励まし**を含み、**オープニング／中盤／終盤**のセグメントを持つこと。読み上げは **Web Speech API** や**短いプリレコード／ビープ**でも可（課題デモのコスト方針）。  
- チェックポイント接近時に、事前に紐づけた**ご褒美・一言**を挿入できること（疑似距離で「接近」を判定してよい）。  
- 他アプリのオーディオ再生と**競合しないよう**、ダッキングまたは同等の**音量・割り込みポリシー**を定義・実装すること。**Web デモ**では OS レベルダッキングは前提とせず、**別タブ音声との併用時の注意表示**や、ユーザー操作によるミュート切替など**ブラウザ範囲の簡易対応**で足りる。

### F-3 走後

- セッションの**サマリー**（距離、時間、平均ペース等）を表示すること。  
- **次回セッションの提案**（例：リカバリー、強度維持）を提示すること。MVP は**ルールベース**で可。 [仮]

### F-4 設定

- 区切り基準（距離／時間）の変更、（Should）区切り粒度、（Phase 2 候補）話量モードの設定を提供すること。 [仮]

### F-5 連携（将来）

- Google Fit 等との**連携インターフェースを拡張可能**にすること。MVP では**設定画面のフック**または未接続のスタブでも可。 [仮]

### 対象外（明示）

- 食べログ連動の飲食紹介。  
- 特定ウェアラブル（Garmin 等）を必須とする機能。  
- ソーシャル・チームラン機能（MVP）。

## 非機能要件

| 区分 | 要件 | 根拠・仮定 |
|------|------|------------|
| 可用性 | モバイルクライアントはオフライン時に**最低限セッション記録と音声キャッシュ**の方針を検討する（完全オフラインは要定義） | 屋外通信不安定 |
| パフォーマンス | フォアグラウンド走行中、**主要操作の応答**は体感で待たせない（目安 1 秒以内、要計測） | 仮定 |
| バッテリー | バックグラウンド位置取得は OS ガイドラインに従い、**更新間隔・精度を段階設定**可能にする [仮] | |
| セキュリティ・プライバシー | 位置・健康関連データは**利用目的の明示、同意、保存期間、削除**をポリシー化する。通信は TLS。 | 個人情報保護法・ガイドライン準拠を目指す |
| 品質（音声） | **自然な人間的な話し言葉**に限りなく近づけることを最優先の体験品質とする（測定は主観評価＋エンジニアリング指標の併用を検討） | プロダクト要件 |
| スケーラビリティ | バックエンドを用いる場合、**ステートレス API** とマネージド DB で水平拡張可能な形を推奨 | 初期〜中期 |

## 技術スタック

サンプルは **`demo-implementation/` の Next.js / TypeScript /（任意）Supabase / Vercel** を拡張する想定とする。**版 0.2** は提出デモのため、**有料 API に依存しない**構成を既定とする。

### Web クライアント（版 0.2 の主実装）

- **Next.js（App Router）・TypeScript**：計画〜走行（疑似）〜完了までの一連 UI。  
- **地図**: **Leaflet**（または react-leaflet）＋ **OpenStreetMap 由来タイル**（利用ポリシー・帰属表示の遵守。タイルプロバイダはプロジェクトで明示する）。  
- **施設・ルート補助データ**: **静的 JSON**（リポジトリ同梱）またはクライアント内モック。本番用の外部 Places API はデモでは不要。  
- **音声**: **Web Speech API（SpeechSynthesis）**、または短い **HTMLAudioElement** 用ダミークリップ。  
- **ホスティング**: Vercel 等（デモ URL 提出が求められる場合に備える）。 [仮]

### バックエンド（デモで任意）

- **Supabase（PostgreSQL）** または**ローカル／メモリのみ**：課題が「フロントのみ」でよい場合は永続化を省略可。 [仮]  
- **認証**: デモでは未ログインでも可。必要時のみ Clerk / Supabase Auth。 [仮]

### モバイルクライアント（将来拡張の候補）

- **React Native（Expo）** または **Swift / Kotlin ネイティブ**：実走・バックグラウンド GPS・オーディオセッション・他アプリ音声との調整。  
- 言語: TypeScript（RN）または各ネイティブ。

### 外部サービス（将来・本番想定）

- **有料地図・施設 API**（例：Google Maps Platform）：コスト・ライセンス検証後に検討。 [仮]  
- **クラウド TTS／収録パイプライン**：品質向上フェーズで検討。 [仮]

**選定理由の要約（0.2）**: 課題の**ブラウザデモ・低コスト**方針に合わせ、地図・POI・走行を**オープン／モック寄り**に寄せる。実走向けのネイティブ／RN はロードマップ上の次段として残す。

## UI/UXの方向性

- **デザインコンセプト**: 走前は計画が明快、走中は**音声主役・画面は最小**、走後は達成感と次の一歩。  
- **トーン＆マナー**: 伴走するラジオのような親しみと、信頼できる数値のバランス。  
- **ターゲットデバイス**: **版 0.2** は **デスクトップブラウザ**での発表・録画を前提にしつつ、**幅狭ビュー**でも破綻しないレスポンシブを推奨（将来のモバイル本番に接続しやすい）。  
- **参考サイト/アプリ**: 主要ランニングアプリ（記録 UI の慣習）、ポッドキャストアプリ（バックグラウンド音声の扱い）— URL は個別に選定。 [仮]

## 追加情報リクエスト

以下を優先的に確定すると、システム要件の精度が上がる。

| 優先度 | 項目 | 影響 |
|--------|------|------|
| 高 | **課題必須スタック**（Web のみ可か、Expo 実機必須か）— 大学・主催者ルール | 0.2 は Web デモ前提だが確定が必要 |
| 高 | **ルート作成方式**（デモではプリセットルート＋地図表示でも可） | F-1 の画面・データ設計 |
| 中 | **本番想定の地図・施設 API**（有料枠・キー管理）— デモ後の拡張 | F-1 の実現性・コスト |
| 中 | **認証・データ同期**の要否 | バックエンド範囲 |
| 中 | **音声方式**（TTS ベンダー、収録範囲） | 品質・スケジュール |
| 低 | 収益モデル・KPI 数値 | ロードマップ・計測実装 |
