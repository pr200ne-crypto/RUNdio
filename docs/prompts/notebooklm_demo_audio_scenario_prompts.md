# NotebookLM 取り込み用：RUNdio デモ音声マスタープロンプト

本ファイルは次の2部で構成する。

1. **§1 運用メモ**（人間向け・NotebookLM には載せない）  
2. **§2 マスター仕様書**（**§2 全体をコピー**して NotebookLM のソースに追加する）  
3. **§3 カスタム指示文**（NotebookLM のチャット／音声カスタマイズ欄に貼る）

---

## §1 運用メモ（NotebookLM にアップロードしない）

1. **§2** を Google ドキュメントに貼って追加するか、NotebookLM で「ソース」としてテキストを追加する。  
2. **§3** を、チャット初回または Audio Overview のカスタム指示に貼る。  
3. デモの数値（距離・ペース・地名・施設名）は **§2 の「4. 本エピソードの固定条件」** だけ差し替えてよい。  
4. 章を分けたい場合は、§2 を複製し「4. 本エピソードの固定条件」だけパート用に書き換え、ソースを分ける。

## §2 マスター仕様書（ここから下をすべて NotebookLM に読み込ませる）

```
================================================================================
文書ID: RUNDIO-NBLM-AUDIO-SPEC-001
文書種別: 音声コンテンツ生成用 マスター仕様書（課題デモ向け）
対象サービス: RUNdio（日本語表記: ランディオ）
キャッチコピー: ラン×レディオ：あなただけのランニング用ラジオ
想定用途: NotebookLM による会話調オーディオ（Audio Overview 等）の内容設計
言語: 日本語（生成物の台本および想定発話はすべて日本語）
================================================================================

1. 目的と位置づけ

1.1 本仕様書は、ソロランナー向けアプリ「RUNdio（ランディオ）」の課題デモにおいて、
    走行中に再生する「2人組の伴走ラジオ」風オーディオの内容を一貫して設計するための
    制約条件と成果物定義を示す。

1.2 本オーディオは医療機器・コーチング資格を前提としない。健康増進・体験説明用の
    フィクションに近いトーンとする。

1.3 ラジオ番組としての体裁（必須・最優先のトーン規定）

    MUST: 生成する会話は、一貫して「放送されているラジオ番組」として聞こえること。
         番組の空気感（オープニングの勢い、ジングや番組名に相当する決まり文句、
         リスナー＝一人のランナーへの呼びかけ、締めの定型など）を、自然な形で含める。
         数値や事実は「説明の箇条書き」ではなく、掛け合い・ツッコミ・間の中に織り込む。

    MUST NOT: アプリの使い方やランニング一般を「ただ説明する会話」にすること。
         講義調・マニュアル読み上げ・インタビュアーと専門家の解説だけの体裁を主としない。
         ニュース原稿のような淡い朗読調や、Podcast の要約だけを淡々と話す調子も避ける。

1.4 スコープ（本仕様が対象とするもの）

    本仕様書が規定するのは「音声として再生される台本・掛け合い」に限る。
    Web やアプリの画面デザイン、UI 文言、サービス全体のビジュアルデザインは対象外とし、
    台本に「画面をこうする」「ボタンが…」等のサービスデザイン指示を書き込まないこと。

1.5 オープニング（冒頭）のラジオ番組化（必須）

    MUST: 台本の最初の発話から、聴取者が「ラジオ番組が始まった」と感じられること。
         次のうち少なくとも2つを、自然な掛け合いの中に含める（表現は創作してよい）。
         - 放送開始に相当する合図（例: 局名・番組名・「それではお聴きください」に相当する一言、
           短いカウントやテンションの上がる一言など。実在の放送局名の借用は不要）
         - リスナー＝走る一人への呼びかけ（例: 今夜も同じ時間に、など）
         - 本編に入る前の「間」や相槌を含め、説明が始まる前に番組の空気を作る

    MUST NOT: 第1発話からいきなり「今日は10kmです」「このアプリでは」など、
         プラン説明・機能説明だけでオープニングを始めること。

2. 用語定義

2.1 「ホストA」「ホストB」: 本オーディオに登場する話者2名。性別・年齢は固定しないが、
    役割分担は第3章に従う。

2.2 「チェックポイント」: ユーザーが走前に設定した、ルート上の目印地点（本エピソードでは
    入浴施設を想定）。

2.3 「区切り」: 距離または時間に基づきアプリが音声イベントを発火する単位。本仕様では
    距離ベースを主とする。

3. 話者仕様（必須）

3.1 ホストA（進行役）
    MUST: 距離・経過時間・ペース等の数値を短く読み上げ可能な形で伝える。
    MUST: 番組の導入・区切り・締めのフレーズを担当する。
    MUST NOT: 聞き手を侮辱・脅す表現を用いる。

3.2 ホストB（相棒役）
    MUST: 共感、軽いツッコミ、ユーモアを挿入する。
    MUST: 「ゴール後のご褒美」の先取りトーク（ワクワクと安心のバランス）を担当する。
    MUST NOT: 上から目線の説教、毒舌、ネガティブな人格付けを主軸にしない。

3.3 掛け合い
    MUST: 相槌・間・短い重なりを含め、実在の対談ラジオに近い口語体とする。
    MUST NOT: 1話者の独白が連続して90秒を超える構成を主としない（デモでは区切りを挟む）。

3.4 ラジオらしさの具体化
    MUST: どこかのタイミングで「今この番組を聴きながら走っている」ことが伝わる言い回しを
         少なくとも1回は入れる（例: イヤホンの向こう、一緒にこのコースを進もう、など。
         文言は仕様に合わせて変えてよい）。
    MUST: 説明が必要な箇所は、必ず相手への一言・リアクション・短い突っ込みで割り込ませ、
         会話のリズムを維持する。

4. 本エピソードの固定条件（差し替え可能項目）

    4.1 想定ユーザー: 平日夜ラン、イヤホン利用、目標距離 10 km。
    4.2 ルート概要: 河川敷を折り返すアウトバックコース。
    4.3 チェックポイント: ゴール付近の入浴施設（固有名はフィクション「〇〇湯」）。
    4.4 区切り: 距離ベース。おおよそ 3 km ごとにトークの節目を置く想定。
    4.5 参考ペース（フィクション）: 平均 6:00/km 前後を想定するが、断定は避け、
         「このくらいの感じ」程度の言及にとどめる。

5. コンテンツ要件

5.1 MUST（必ず満たすこと）
    - 全体の聞こえ方が「ラジオ番組」であり、単なる機能説明や一般論の対談に見えないこと
      （仕様書 1.3・3.4 に適合すること）。
    - ホストAおよびホストBの2名が登場する。
    - 仕様書 1.5 に従い、冒頭がラジオのオープニングとして成立していること。
    - オープニング（第1章）の中で「今日の距離・コースの雰囲気・チェックポイント（銭湯）」が
      分かる（ただし 1.5 の MUST NOT に反しない順序で述べること）。
    - 中盤で 3 km / 6 km 前後の節目を意識した会話がある。
    - チェックポイント接近時に、ご褒美の先取りトークがある。
    - 終盤で残り距離が少ない文脈の会話と、気持ちのよい締めがある。
    - サービス名「RUNdio」または「ランディオ」を、自然な文脈で合計 1〜2 回まで言及する。
    - 日本語のみで記述・発話する。

5.2 SHOULD（満たすことが望ましいこと）
    - 各パートで「聞き手が一人のランナーである」前提が透けて見える。
    - 次回セッションへの軽い言及（例: 明日は短め、など）を会話に含める（医療判断ではない）。

5.3 MUST NOT（禁止）
    - アプリや走り方を「解説するだけ」の会話に終始すること（ラジオ番組としての掛け合い・
      感情・リズムが主役でない成果物にしない）。
    - 病名の診断、治療指示、薬の勧誘。
    - 極端な過負荷を推奨する表現（「限界まで」「絶対に止まるな」等）。
    - 実在の特定事業者に対する誹謗中傷、または根拠のない優劣の断定。
    - ブランド名の連呼（スポンサー読み上げのような反復）。
    - 政治・宗教の勧誘。

6. 構成仕様（章立て）

    第1章 オープニング（想定 60〜120 秒相当の情報量）
        必ず仕様書 1.5 の順序感を守る: 先に「放送が始まった」体裁（局／番組／リスナーへの声かけ等）、
        その後に今日のプランの要約、2人の自己紹介に相当する軽いやりとり。
        プラン内容の説明は、オープニングの空気が立ってから入れる。

    第2章 走行前半〜中盤（3 km 付近を含む）
        リズム・ペースの話、軽い励まし。

    第3章 走行中盤（6 km 付近を含む）
        少しだけテンションの変化（まだいける／息を整える等）、ユーモア1つ。

    第4章 チェックポイント接近
        銭湯「〇〇湯」への期待、ゴールが見えてきた感覚。

    第5章 フィニッシュ〜エンディング（想定 60〜120 秒相当の情報量）
        完了の祝福、次の一回へのさりげない提案、締めの挨拶。

7. 台本フォーマット（NotebookLM が中間成果物を出す場合の推奨形式）

    各発話を次の形式で列挙する。
    [話者: ホストA または ホストB] セリフ（口語体）

    必要に応じて括弧でト書きを付けてよい。
    例: （軽く笑いながら） など。ただしト書きは最小限とする。

8. 尺・密度

8.1 本エピソード全体の想定は、デモ用に情報量として 5〜8 分の会話に相当する密度とする。
8.2 実際の再生尺は生成エンジンに依存するため、本仕様は「情報量」と「章比率」を規定する。

9. 著作・利用上の注意（人間オペレータ向けメモ）

    生成物の利用・再配布は NotebookLM および利用環境の利用規約に従うこと。
    本仕様書自体はプロジェクト内部のデモ用資料とする。

================================================================================
END OF SPECIFICATION
================================================================================
```

## §3 カスタム指示文（NotebookLM のチャット／音声用。§2 をソース追加したあとに貼る）

```
あなたは上記マスター仕様書（RUNDIO-NBLM-AUDIO-SPEC-001）に厳密に従う編集者兼台本作家である。
アップロード済みの仕様書のみを根拠に、次を実行せよ。

(0) 成果物は音声向けの台本のみとする。Web・アプリの画面デザインや UI の指示は台本に含めない
    （仕様書 1.4）。ラジオらしさはセリフと掛け合いだけで表現する。
(0b) 第1章の最初の数往復は、仕様書 1.5 に従い「番組の始まり」として明確に聞こえるように書く。
     いきなり距離や機能の説明から入らない。

(1) 第1章から第5章までの会話台本を、仕様書 7 章のフォーマットで全文出力する。
(2) ホストA・ホストB のセリフ比率が極端に偏らないようにせよ。
(3) 数値は仕様書 4 章の固定条件に合わせよ。追加の統計や事実は捏造しない。
(4) 仕様書 5.3 の MUST NOT に該当する表現を含めない。
(5) 台本は「ラジオ番組として放送している」前提で書くこと。説明口調・講義・マニュアル調の
    連続を避け、必ず番組らしい掛け合い・間・リスナーへの声かけを含めること。
(6) もし台本が「ただの説明の会話」に寄りかけていると自己評価したら、該当箇所を特定し、
    ラジオ的なト書き・ツッコミ・フレーズに差し替えた改稿版を出力すること。

その後、Audio Overview（または同等の会話調音声）向けに、
(7) 各章の要約を1〜2文で付し、音声の聞きどころ（感情のピーク）を箇条書きで示せ。
```
