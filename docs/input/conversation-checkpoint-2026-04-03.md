# 対話チェックポイント（ランディオ）

**日付**: 2026-04-03  
**モード**: A（自分で話すモード）  
**目的**: ビジネス／プロダクト要件のたたき台を言語化し、後続の詳細要件定義に引き継ぐ。

---

## 1. プロダクトの核

| 項目 | 内容 |
|------|------|
| 名称 | **ランディオ** |
| キャッチコピー | **ラン×レディオ：あなただけのランニング用ラジオ** |
| 一言 | **あなた専用のランニング用ラジオ**が、走前から走後まで一本の番組のように寄り添う |
| 解決したいこと | 一人ランの**マンネリ**、**モチベーション維持**、走行中の**楽しさ** |

---

## 2. ユーザー課題・体験仮説

- いつも一人で走っており、**練習内容がマンネリ化**しやすい。
- 走行中は**音声で寄り添ってほしい**（数値フィードバックに加え、**励まし**も欲しい）。
- 走ったあとは**振り返り**と、**次のセッションの提案**（例：強度を上げる／リカバリーラン）が欲しい。
- **ご褒美の先取り**でモチベが上がる例：ゴール近くが銭湯なら「終わったらお風呂、冷えたビールが待っている」など。

---

## 3. スコープの明確化

### 3.1 含める（合意）

- **走前の「今日の番組」づくり（プランニング）**: 距離・ゴールイメージ・**ルート上の条件検索**（例：「このルートの先の入浴施設」）で今日の一本を決めやすくする。
- **走行中の音声体験（最優先）**: ペース・心拍などの数値 ＋ 励まし。**ラジオ的な番組構成**（オープニング／中盤／終盤など）と、**ラジオ・音楽との共存**（ダッキング、短い一言から、など）を重視。
- **言い回しのバリエーション**、**話量・頻度モード**（例：黙々／標準／よしよし多め）のような**ランナー体験の工夫**。
- **通知・区切りの基準**: ランナーが **距離ベース** と **時間ベース** の**好きな方を選択**可能にする。
- **自然で人が話しているような音声**を最大限の目標に（品質方針として記録）。
- **スマホ単体で利用可能**。Garmin 等は**前提にしない**（ユーザーは Garmin 併用可）。
- **Google Fit 等との連携**: 詳細は未決。**拡張の余地を残す**方針。

### 3.2 明示的に除外・撤回

- **食べログ連動・店紹介**（外部レビューサイトに依存するグルメ案内）は**採用しない**。対話内で撤回済み。

---

## 4. ルート・検索まわり

- **「10km先」等の基準**は **ルート上**（直線距離だけではなく、**今日走るルートに沿った距離軸**）。
- **最初の主役は条件検索**（手動ピン留めだけだと初回ハードルが高い）。ピン留めは**補助**として残す余地あり。
- 過去 README の **提案 A（希望→コース自動生成型）** と親和性が高い。B/C は将来拡張候補。

---

## 5. 音声・UX（合意した方向性）

- 重視したい柱: **番組構成**、**他オーディオとの被せ方**、**言い換えプール**、**ユーザーが選べる話量・頻度**。
- キャラ（厳しめコーチ／DJ 等）は**未決** — 詳細要件で確定予定。

---

## 6. 未決・要フォロー（詳細要件で詰める）

- ターゲットの幅（自分以外のペルソナ）、**差別化・競合**、**Why Now / Why You**、**ビジネスモデル**、**KPI**。
- ルート作成 UI（自動ルート生成の粒度、GPX 等の扱い）。
- 音声実装方針（TTS のみ／収録＋TTS ハイブリッド等）— 技術選定は別プロンプトでも可。
- Google Fit 連携の**具体的な読み書き範囲**（将来）。
