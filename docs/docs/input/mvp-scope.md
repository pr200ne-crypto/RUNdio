# MVP 範囲（ランディオ）

## MVP 定義

### 目的

ソロランナーに対し、**ルート上条件検索による計画**と**走行中のパーソナル音声体験**、**走後の次回提案**の価値を**最小構成で検証**する。 [仮]

### 対象ユーザー

主ペルソナに近い**週数回ソロランするスマホユーザー**。 [仮]

### 成功の定義（例）

- 走行中に**主観的な満足・伴走感**が得られたというフィードバック。  
- **セッション完了率**が一定以上。 [仮・数値は未定]

---

## スコープ

### MVP に含める ✅

| 領域 | 内容 |
|------|------|
| ルート | アプリ上で**今日のルート**を定義できる（方式は実装で決定） |
| 検索 | **ルート上の前方**を基準にした**条件検索**（少なくとも1カテゴリ、例：入浴施設） [仮] |
| 音声 | **数値＋励まし**、**オープニング／中盤／終盤**の粗い番組構成 |
| 区切り | ユーザーが**距離**または**時間**を選択 |
| 共存 | **他オーディオとの被せ**の基本方針（ダッキング等）を実装または検証可能な形で |
| 走後 | **サマリー**＋**次回提案**（ルールベースから開始可） [仮] |

### MVP に含めない ❌（初期）

| 内容 | 理由 | 対応時期 |
|------|------|----------|
| 食べログ系グルメ連携 | スコープ外 | 実施しない |
| Google Fit の本実装 | 拡張フックのみで可 | Phase 2 |
| 多キャラ音声・高度な生成 | コスト・複雑さ | Phase 2 |
| ソーシャル・チーム機能 | コアと乖離 | 要評価 |

## 制約事項

- 地図・施設データの**ライセンス・コスト**。  
- **バッテリー・バックグラウンド**制約（OS 差）。  
- 音声の**自然さ**と**開発コスト**のトレードオフ。

---

## リスクと対策

| リスク | 対策 |
|--------|------|
| ルート上距離と検索のずれ | 早期プロトタイプで精度検証 |
| 音声が不自然 | 短文・ハイブリッド収録＋TTS の試作 |
| MVP が肥大化 | 本ドキュメントの Must に厳守 |
